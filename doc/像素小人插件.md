# 像素小人（2D Avatar）插件落地方案（使用免费资源）

本文目标：在 **ProactiveAI Desktop** 内落地一个“像素虚拟形象”插件（后续可作为插件市场核心差异化）。本方案 **不自绘**，使用可商用的免费像素资源（优先 **CC0**），并把资源标准化为可分发的 `avatar-pack` 包。

> 重要约束：桌面端必须 **离线可用**；不引入远端渲染与第三方广告 SDK；资源许可可追溯、可在应用内展示 Credits。

---

## 1. 非目标（明确不做）

- 不做 3D（VRM/Live2D 等）；
- 不做网络同步“云皮肤”；
- 不做摄像头驱动、语音驱动、全身动作捕捉；
- 不做第三方插件自定义 React 组件（先用 **声明式 manifest** + 资源包）。

---

## 2. 免费像素资源选择（可商用）

本插件建议优先使用 **CC0** 资源（免署名、可商用），避免后续分发时因署名/传染性条款引发问题。

### 2.1 推荐来源

- **Kenney**：大量游戏素材包，常见为 **CC0/宽松许可**（以具体包声明为准）。优点：风格统一、打包完整、可快速做出“可卖”体验。
- **OpenGameArt**：资源多但许可混杂（CC0/CC-BY/GPL）。**仅选 CC0** 或明确可商用许可；若为 CC-BY 必须在应用内展示署名与链接信息。
- **itch.io**：可按“CC0 / public domain”筛选，仍需逐包核对 license 文本。

### 2.2 许可落地规则（硬性）

- 每个 `avatar-pack` 必须包含 `LICENSE.txt` 与 `CREDITS.json`（见 §4.4）。
- 只允许以下许可进入“官方内置/推荐”：
  - `CC0-1.0`
  - `MIT`（少见于素材，但可接受）
  - `CC-BY-4.0`（可接受，但必须显示署名；默认不作为内置首选）
- 禁止：GPL/AGPL/CC-BY-SA 等可能影响整体分发的许可。

## 3. 用户体验（UX）定义

### 3.1 形象展示位置（一期）

- 在 **对话主窗口**右下角叠加一个“AvatarWidget”（不影响聊天布局）。
- 支持拖拽移动与吸附边缘（可选：二期）。
- 支持最小化/隐藏（设置里开关）。

### 3.2 控制方式（一期）

1) **斜杠指令（本地解析，不发给模型）**

- `/avatar show|hide`
- `/avatar mood idle|happy|angry|sad|confused`
- `/avatar emote blink|sweat|heart`（短暂表情叠加）
- `/avatar pack list`
- `/avatar pack use <packId>`

2) **事件联动（可选，低风险）**

- 收到 AI 回复（assistant message）触发 `avatar:react`（例如包含 `?` → confused 1s）
- 触发点触发（proactive trigger）触发 `avatar:react`（例如“关心” → happy）

> 一期建议只做 1) 指令，2) 联动先用简单规则，避免误判引起“角色乱动”。

## 4. 资源包格式（avatar-pack）

### 4.1 文件扩展名与解压策略

- 扩展名：`.pavatar`（ProactiveAI Avatar Pack）
- 本质：ZIP
- 解压位置：`userData/data/avatars/packs/<packId>@<version>/`

### 4.2 包结构（固定）

```text
pack.pavatar
├── manifest.json
├── sprites.png
├── animations.json
├── preview.png                 # 可选：用于市场卡片预览
├── LICENSE.txt                 # 必填
└── CREDITS.json                # 必填
```

### 4.3 `manifest.json`（schema_version=1）

```json
{
  "schema_version": 1,
  "pack_id": "com.proactiveai.avatars.kawaii-cc0",
  "name": { "zh-CN": "可爱像素小人", "en-US": "Kawaii Pixel Avatar" },
  "author": "Kenney",
  "version": "1.0.0",
  "license": "CC0-1.0",
  "description": {
    "zh-CN": "支持 idle/happy/angry/blink 等表情。",
    "en-US": "Includes idle/happy/angry/blink animations."
  },
  "sprite": {
    "frame_width": 64,
    "frame_height": 64,
    "scale": 3
  },
  "entry_animation": "idle"
}
```

### 4.4 `CREDITS.json`（必填）

```json
{
  "sources": [
    {
      "name": "Kenney",
      "license": "CC0-1.0",
      "homepage": "https://kenney.nl/",
      "asset_pack": "（填写具体包名或页面）"
    }
  ]
}
```

> 如果来源为 CC-BY：必须包含作者署名与原始链接，并在 UI 中展示。

### 4.5 `animations.json`

建议采用“状态机 + 帧序列”：

```json
{
  "animations": {
    "idle": { "frames": [0,1,2,3], "fps": 6, "loop": true, "priority": 0 },
    "happy": { "frames": [16,17,18,19], "fps": 10, "loop": true, "priority": 5 },
    "angry": { "frames": [32,33,34,35], "fps": 10, "loop": true, "priority": 5 },
    "blink": { "frames": [48,49,50], "fps": 12, "loop": false, "priority": 9, "duration_ms": 400 }
  }
}
```

实现约束（硬性）：

- `priority` 数字越大越“强制”，可打断低优先级动画；
- `loop=false` 的动画结束后自动回退到 `idle`（或 `entry_animation`）；
- 不允许引用不存在的帧索引；加载时必须校验。

## 5. 技术实现（工程落地）

## 5.0 执行模型：主进程 IPC 调度 + 渲染进程 Worker（默认方案）

结论（硬性约定）：

- **主进程**：只负责“低频、需要权限与安全边界”的事情：安装/卸载资源包、持久化 settings、提供 packs 列表、做 ZIP Slip/许可校验。  
- **渲染进程**：负责 UI 展示与 WebGL 渲染（Pixi 必须在渲染进程）。  
- **AvatarEngine**：运行在 **渲染进程的 WebWorker** 中，负责指令解析、动画状态机、优先级与打断等逻辑。  

这样可以避免高频动画状态（每帧）跨 Electron IPC 往返，既不会卡 UI，也不会让主进程变成瓶颈。

整体通信分两层：

1) **renderer ↔ main（Electron IPC）**：低频（安装/卸载/设置）  
2) **renderer UI ↔ AvatarEngine Worker（postMessage）**：中频（指令/事件/状态）  

> 注：如果未来需要更强隔离（例如“远端技能”“付费插件运行时”），再升级为主进程子进程/worker_threads，但 Pixi 渲染仍必须在 renderer。

### 5.1 渲染技术栈

- 渲染进程：**PixiJS**（建议 `pixi.js`）
- 画布容器：React 组件 `AvatarWidget`
- 像素清晰：启用 nearest-neighbor（pixi texture scale mode = NEAREST），避免模糊。

### 5.2 代码位置建议（与现有结构对齐）

```
src/renderer/components/AvatarWidget/
  index.tsx
src/renderer/avatar/
  pack-types.ts
  pack-loader.ts
  avatar-state-machine.ts
  command-parser.ts
src/main/avatar-pack-store.ts
src/main/index.ts  # IPC 挂载
```

### 5.3 IPC（主进程）

> 注意：资源安装/解压/持久化必须在主进程完成；渲染进程只负责展示与指令 UI。

建议 IPC：

| 通道 | 方向 | 参数 | 返回 | 说明 |
| --- | --- | --- | --- | --- |
| `avatar:packs:list` | renderer→main | - | `AvatarPackSummary[]` | 已安装包列表 |
| `avatar:packs:installFromFile` | renderer→main | `filePath` | `{ok,error?}` | 安装 `.pavatar`（ZIP Slip 防护） |
| `avatar:packs:remove` | renderer→main | `packId,version` | boolean | 删除包 |
| `avatar:settings:get` | renderer→main | - | `AvatarSettings` | 当前使用包、开关、位置 |
| `avatar:settings:set` | renderer→main | `partial` | boolean | 保存配置 |

### 5.3.1 AvatarEngine Worker（渲染进程）消息协议（schema_version=1）

Worker 文件建议：`src/renderer/avatar/engine.worker.ts`

#### Worker 输入消息（UI → Worker）

```ts
type AvatarEngineIn =
  | { v: 1; type: 'INIT'; pack: AvatarPackResolved; settings: AvatarSettings }
  | { v: 1; type: 'SET_MOOD'; mood: string }
  | { v: 1; type: 'PLAY_EMOTE'; name: string }
  | { v: 1; type: 'TICK'; now_ms: number }
  | { v: 1; type: 'ON_ASSISTANT_MESSAGE'; text: string }
  | { v: 1; type: 'ON_TRIGGER'; text: string }
```

说明：

- `TICK` 由 UI 定时发送（例如每 50ms 或 `requestAnimationFrame` 采样）用于驱动非循环动画到期回退；也可完全由 Worker 内部 `setInterval` 驱动（推荐 UI 侧驱动，便于统一暂停/销毁）。
- `ON_*` 事件用于“低风险联动”（可在设置中开关）。

#### Worker 输出消息（Worker → UI）

```ts
type AvatarEngineOut =
  | { v: 1; type: 'READY' }
  | { v: 1; type: 'STATE'; animation: string; frame: number; overlay?: string }
  | { v: 1; type: 'ERROR'; code: string; message: string }
```

硬性规则：

- `STATE` 为“可渲染状态”，UI 每次收到后只做纯渲染，不做业务判断。
- Worker 任何异常都必须转成 `ERROR`，UI 收到后自动隐藏 widget 并提示错误（不影响聊天主流程）。

#### 超时与恢复（硬性）

- UI 在发送 `INIT` 后等待 `READY`，超时（如 2s）则认为 Worker 启动失败；
- Worker `ERROR` 后，UI 提供“一键重启 AvatarEngine”按钮（可选：二期）。

### 5.4 本地存储

- `avatar-pack-store`：lowdb/JSON 或 electron-store 均可，但建议跟“会话/消息”一样放 `userData/data/`：

```text
userData/data/avatars/
  packs/
    <packId>@<version>/
      manifest.json
      sprites.png
      animations.json
      LICENSE.txt
      CREDITS.json
  avatar-settings.json
```

`avatar-settings.json` 示例：

```json
{
  "enabled": true,
  "active_pack": { "pack_id": "com.proactiveai.avatars.kawaii-cc0", "version": "1.0.0" },
  "position": { "x": 980, "y": 620 },
  "scale": 3
}
```

## 6. 指令解析与消息流（关键细节）

### 6.1 斜杠指令必须在本地拦截

规则（硬性）：

- 以 `/avatar ` 开头的输入 **不发送给模型**；
- 在 UI 中作为“系统操作提示”或“本地消息”展示（例如 role=`assistant` 的本地提示），避免对话上下文污染。

落地点：`src/renderer/components/InputArea/index.tsx` 的 `handleSend` 前置解析（或者抽到 `command-parser.ts`）。

### 6.2 指令到 AvatarWidget 的通信

建议流程：InputArea 解析指令 → 调用 AvatarWidget 暴露的 `dispatch` → `postMessage` 给 Worker：

- `worker.postMessage({ v: 1, type: 'SET_MOOD', mood: 'happy' })`
- `worker.postMessage({ v: 1, type: 'PLAY_EMOTE', name: 'blink' })`

## 7. 市场卡片展示字段（为后续插件市场做准备）

即使一期不做远程市场，也要把“卡片展示字段”固定在 `manifest.json`：

- `name`（多语言）
- `description`（多语言）
- `author`
- `version`
- `license`
- `preview.png`

卡片 UI 不应显示内部路径/ID；高级信息可折叠展示 `pack_id` 与 license 文本摘要。

## 8. 安全与质量清单（必须实现）

- ZIP Slip 防护：解压每个 entry 前做 `path.normalize`，必须在目标目录前缀内。
- 大小限制：包大小（如 10MB）与解压后总大小（如 50MB）。
- 资源校验：manifest schema_version、frame 尺寸、animations 引用帧索引合法。
- 许可校验：`LICENSE.txt` 与 `CREDITS.json` 必须存在；license 不在白名单则拒绝“官方推荐”，但可允许用户手动安装并提示风险（产品策略）。
- 崩溃隔离：Pixi 初始化失败不影响聊天主流程，Widget 自动隐藏并提示错误。

## 9. 里程碑与验收（工程可落地）

### M1：渲染 + 内置 demo 资源包（不带安装）

- 固定从 `src/renderer/assets/avatar-demo/` 加载 1 套 CC0 资源（先写死路径）
- `AvatarWidget` 可在右下角播放 `idle`
- 指令 `/avatar mood happy` 可切换动画并回 idle

**验收**：在 Windows 上帧率稳定、像素不糊（nearest-neighbor 生效）。

### M2：`.pavatar` 安装/卸载 + packs:list

- 主进程安装 ZIP → 写入 `userData/data/avatars/packs`
- UI 可切换 pack，并持久化

**验收**：重启后仍保持上次使用的 pack。

### M3：卡片式“Avatar 皮肤库”页面（本地）

- 展示已安装 packs 的卡片（preview、作者、版本、许可）
- 提供导入本地 `.pavatar` 按钮

**验收**：安装失败有明确错误码与提示（缺文件、许可缺失、动画索引越界）。

## 10. 与插件系统的关系（建议）

当前仓库的通用插件系统（hooks/marketplace）处于反复调整阶段。建议像素小人先作为 **“第一优先官方插件”**独立落地：

- 资源包与 UI 先落地并稳定协议；
- 再把其抽象成通用插件市场的一种 `PluginType = avatar-pack`；
- 最终市场可以同时卖：`avatar-pack`（皮肤/动作）与 `hook-plugin`（逻辑扩展）。

这样不会因为通用插件协议未定而阻塞“可卖的差异化体验”上线。
